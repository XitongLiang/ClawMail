## V2 — JSON 解析失败暴露

更换部分测试邮件后，出现 JSON 解析失败。

### 发现的问题

| # | 邮件 | 失败原因 |
|---|------|---------|
| #5 | BMY 帖子 | LLM 用 ` ```json ``` ` 代码块包裹输出，内部 JSON 还是畸形的 |
| #7 | 法律帖 | LLM 完全无视 JSON 格式要求，返回 markdown 分析 |

### 修复：多层 JSON 解析防御

```
LLM 输出
  → json.loads() 首次尝试
  → _strip_code_fences()     # 去除 ```json...``` 包裹
  → 提取 JSON 片段 ({...} 或 [...])
  → _repair_json_quotes()    # 修复值内未转义引号
  → 重试（加 "strict JSON only" 提示）
```

新增函数：
- `_strip_code_fences(text)` — regex 去除代码块包裹
- `_repair_json_quotes(text)` — 修复中文语境下的嵌套引号

## V3 — 加 response_format

### 发现的问题

V2 修复后测 10 封：9/10 成功，#8（法律规定）仍失败。
原因：LLM 在 JSON 值内使用了未转义的中文引号 `起诉"排除妨碍"强制搬离`。

### 修复

在 `call_llm()` 中加入：

```python
"response_format": {"type": "json_object"}
```

强制 kimi-k2.5 输出 JSON 模式，从源头减少格式问题。

### V4 验证（#1-10）

10/10 全部成功。JSON 解析问题基本解决。

## V5 — 质量改进三件套

对 #11-20 批次进行测试后，发现三类质量问题并逐一修复。

### P0: brief 过长

**问题**：prompt 说"不超过150字"，但 LLM 经常超。

**修复**：
- prompt 改为 `"2-3行纯内容摘要（硬限100字）"`
- Python 后处理截断（后来移除，见 V6）

### P1: brief 元评论

**问题**：brief 中出现 `"这是一封论坛转帖邮件"` `"属于XX性质"` `"无需回复"` 等元评论。

**修复**：在 prompt 的防幻觉规则中加入：

```
brief 只写邮件**说了什么**（事实内容），
禁止写"这是一封XX类邮件"、"属于XX性质"、"无需回复"等元评论
```

同步更新了 `references/prompts/summary_guide.md`。

### P2: 内容安全过滤崩溃

**问题**：#16（抵制日货话题）触发 kimi-k2.5 内容安全过滤，返回 400 错误，导致 pipeline 崩溃。

**修复**：
- 新增 `ContentFilterError` 异常类
- `call_llm()` 中捕获两种情况：
  - HTTP 400 + body 含 "high risk"
  - HTTP 200 但 response content 含 "rejected" + "high risk"（kimi 特有）
- `analyze_email()` 中 catch 后返回占位结果：
  - `one_line`: `"（内容无法分析）{subject[:20]}"`
  - `brief`: `"邮件内容触发安全过滤，无法生成摘要。"`

### 验证（#11-20）

| 指标 | V5 结果 |
|------|---------|
| 成功率 | 8/10 OK + 1 content_filtered + 1 json_error（LLM 非确定性） |
| brief 长度 | 65-85 字（prompt 约束生效） |
| 元评论 | 0/9 |

## V6 — 移除 brief 强制截断

### 发现的问题

Python 端对 brief > 100 字的强制截断在实际测试中表现不好。

以 #26（HiAll 招聘）为例：

| | 内容 | 字数 |
|---|------|------|
| **截断前** | …php/html程序员…，美工设计师…，兼职编辑…，**管理员。月薪500-1000元…** | 116 |
| **截断后** | …php/html程序员…，美工设计师…，兼职编辑（负责撰写和采访）。 | 63 |

"管理员"是第四个招聘岗位，被截掉等于丢失核心信息。截断在句号处虽然语法完整，但语义不完整。

### 修复

移除 `_enforce_list_limits()` 中的 brief 截断逻辑，只靠 prompt 约束长度。
LLM 偶尔超出 100 字也不强制截，保证信息完整。

## V7 — 修复 `_repair_json_quotes` 污染 keywords

### 发现的问题

测试 #31-50 时发现 #46（让女孩子哭的分手答案）的 keywords 被污染：

```json
// 期望
"keywords": ["爱情", "分手", "情感故事", "浪漫", "生活哲理"]

// 实际
"keywords": ["爱情「, 」分手「, 」情感故事「, 」浪漫「, 」生活哲理"]
```

5 个关键词变成 1 个拼接字符串。

**根因**：`_repair_json_quotes()` 使用 regex 盲替换，将 `\u201c...\u201d`（中文弯引号）全部替换为 `「...」`。当 LLM 输出混用了弯引号作为 JSON 结构引号时，数组分隔符被一起替换。

### 修复

用**逐字符状态机**替代 regex 盲替换：

```
旧方案（regex）:
  re.sub(r'\u201c(...)\u201d', r'「\1」', text)  # 无法区分结构引号

新方案（状态机）:
  1. 统一 \u201c\u201d → ASCII "
  2. 逐字符扫描，维护 in_string 状态
  3. 遇到 " 时：
     - 后面跟 , ] } : → JSON 结构引号，保留
     - 后面跟其他内容 → 值内嵌套引号，替换为「」
```

### 验证

```python
# Case 1: 值内嵌套引号 → 正确替换
'起诉"排除妨碍"强制搬离'  →  '起诉「排除妨碍」强制搬离'

# Case 2: 正常 keywords 数组 → 不再被污染
["爱情", "分手", "浪漫"]  →  ["爱情", "分手", "浪漫"]  ✓

# Case 3: 弯引号作为 JSON 结构引号 → 正确解析
{"\u201ckeywords\u201d": ["\u201c爱情\u201d"]}  →  {"keywords": ["爱情"]}  ✓
```

## V8 — 预处理（已实现，待真实邮件验证）

### 已实现的预处理函数

| 函数 | 功能 | 匹配模式 |
|------|------|---------|
| `strip_quoted_content(body)` | 去除引用历史 | `>` 前缀行、`On...wrote:`、`在...写道：`、转发头 |
| `strip_signature(body)` | 去除签名块 | RFC 3676 `-- `、`Best regards`、`此致`、`Sent from my iPhone` 等 |

调用位置：`_build_email_user_prompt()` 中，在 4000 字截断**之前**调用。

### 测试集效果

trec06c 测试集主要是论坛帖子，引用格式不标准，仅 3/50 封被清理（14%-28%）。
**真实的 Gmail/Outlook 回复链邮件预计有 30-80% 内容被清理**，这是预处理的主要受益场景。
