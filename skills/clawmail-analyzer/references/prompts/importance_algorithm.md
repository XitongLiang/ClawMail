# 重要性评分规则

## 评分机制

LLM 输出四个维度的原始分（0-100），Python 负责加权计算总分：

```
总分 = sender_score × 0.30 + urgency_score × 0.25 + deadline_score × 0.25 + complexity_score × 0.20
```

**LLM 只需输出 `importance_scores` 中的四个原始分，不需要计算总分。**

所有评分均从**收件人（我）的视角**判断——这封邮件对我有多重要/紧急？

---

## 维度1: sender_score — 发件人对我的重要性 (权重30%)

**优先级：历史记忆 > 职位关键词推测**

### 有发件人画像记忆时，依据记忆中的关系打分：

| 关系类型 | 分数范围 | 说明 |
|---------|---------|------|
| 上司/直接上级/`inbound_command` | 70-95 | 结合过往邮件紧迫性调整 |
| 外部客户/合作方（非命令关系） | 50-70 | |
| 平级同事/`bidirectional` | 40-60 | |
| 下属/`outbound_report` | 30-50 | |
| 纯信息推送/`inbound_info` | 10-30 | |

### 无记忆时，用职位关键词推测：

| 身份类型 | 分数范围 | 判断依据 |
|---------|---------|---------|
| 家人/CEO/总经理/董事会成员 | 90-100 | 邮箱匹配 VIP 列表，或职位关键词 |
| 部门经理/总监 | 70-89 | 直接上级，或部门负责人 |
| 项目经理 | 50-69 | 当前项目相关 |
| 普通同事 | 30-49 | 无特殊职位标识，内部员工 |
| 系统邮件/自动通知 | 0-29 | noreply / notification / system |
| 外部未知 | 10-40 | 根据域名判断 |

---

## 维度2: urgency_score — 邮件对我的紧急程度 (权重25%)

**注意：发件人自身的紧急事务 ≠ 对我紧急。**

| 级别 | 分数 | 信号 |
|-----|------|------|
| 极高 | 90-100 | "紧急"、"立即"、"马上"、"asap"、"urgent"，且要求我执行 |
| 高 | 70-89 | "今天"、"尽快"、"速回"、"priority"，且涉及我的行动 |
| 中 | 50-69 | "本周"、"这几天"、"近期" |
| 低 | 30-49 | "请"、"需要"、"麻烦"，一般性请求 |
| 无 | 0-29 | 无时间相关词汇，或纯信息/分享类邮件 |

知识分享、论坛转帖、个人随笔等非事务性邮件：urgency_score 应为 **0**。

---

## 维度3: deadline_score — 需要我响应的截止时间紧迫性 (权重25%)

**注意：发件人自述的 deadline 不算，只看需要我在什么时间前做出响应。**

| 截止时间 | 分数 | 说明 |
|---------|------|------|
| 今天 | 90-100 | 当日截止 |
| 明天 | 70-89 | 次日截止 |
| 本周内 | 50-69 | 本周日前 |
| 下周 | 30-49 | 下周一至周日 |
| 更晚/无 | 0-29 | 无明确时间或超过一周 |

知识分享、论坛转帖等非事务性邮件：deadline_score 应为 **0**。

---

## 维度4: complexity_score — 需要我处理的任务复杂度 (权重20%)

| 待办特征 | 分数 | 说明 |
|---------|------|------|
| ≥3个高优先级待办 | 90-100 | 大量紧急任务 |
| 2个高优先级待办 | 70-89 | 较多紧急任务 |
| 1个高优或≥2个中优 | 50-69 | 中等工作量 |
| 1-2个低优先级待办 | 30-49 | 少量简单任务 |
| 无明确待办 | 0-29 | 纯通知/讨论/分享 |

---

## 特殊规则

- **is_spam = true** → 不评分，Python 强制 importance_score = 0
- **订阅邮件**（category 含 subscription）→ sender_score 上限 30
- **系统通知邮件** → complexity_score 强制 0（无待办）
